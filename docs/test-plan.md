# 个人信源库 - 测试策略与计划

## 1. 测试策略总览

### 1.1 测试层级

| 层级 | 覆盖目标 | 占比 | 工具建议 |
|------|----------|------|----------|
| 单元测试 | 独立函数、类方法、工具函数 | 60% | Vitest / Jest |
| 集成测试 | 模块间交互、数据库操作、API调用 | 25% | Vitest + Supertest / pytest |
| 端到端测试 | 用户完整工作流 | 10% | Playwright / Cypress |
| 性能/压力测试 | 并发、大数据量、长时间运行 | 5% | k6 / Artillery |

### 1.2 质量标准

- 单元测试覆盖率目标：≥ 80%（核心模块 ≥ 90%）
- 集成测试覆盖所有 API 端点和关键数据流
- 所有 PR 必须通过 CI 测试流水线
- 关键 bug 修复必须附带回归测试用例

| SCH-U06 | 任务优先级排序 | 多个不同优先级任务 | 按优先级排序执行 | P1 |
| SCH-U07 | 任务超时处理 | 超时任务 | 状态变为timeout | P1 |
| SCH-U08 | 任务重试机制 | 失败任务，重试次数<上限 | 重新入队 | P0 |
| SCH-U09 | 任务重试上限 | 失败任务，重试次数=上限 | 状态变为failed | P0 |
| SCH-U10 | 任务递归生成 | 整站爬取发现子链接 | 生成子任务并关联父任务 | P0 |

#### 集成测试

| 编号 | 测试场景 | 验证点 |
|------|----------|--------|
| SCH-I01 | 调度器与爬虫引擎协作 | 任务分发→爬虫执行→结果回传→状态更新 |
| SCH-I02 | 调度器与精炼模块协作 | 爬取完成→触发精炼任务→精炼结果存储 |
| SCH-I03 | 并发任务调度 | 多任务并发执行，无死锁、无资源竞争 |
| SCH-I04 | 任务持久化与恢复 | 服务重启后，未完成任务恢复执行 |

#### 边界条件

- 同时提交 1000+ 任务时的队列行为
- 任务队列满时的拒绝策略
- 调度器崩溃恢复后的任务一致性
- 递归爬取深度限制（防止无限递归）

#### 爬虫测试特殊处理

```
测试策略：
├── Mock HTTP Server（主要）
│   ├── 使用 nock / msw / wiremock 模拟HTTP响应
│   ├── 预设固定HTML内容用于解析测试
│   ├── 模拟各种HTTP状态码（200/301/403/404/500/503）
│   ├── 模拟慢响应（超时测试）
│   └── 模拟动态内容（JavaScript渲染页面）
├── 录制回放（辅助）
│   ├── 使用 polly.js / VCR 录制真实HTTP交互
│   └── 回放用于回归测试
└── 真实网络测试（仅CI定期执行）
    ├── 选取稳定的公开测试站点
    ├── 标记为 @slow / @network 可跳过
    └── 失败不阻塞主流水线
```

#### 边界条件

- 超大页面（>10MB HTML）
- 循环链接检测（A→B→C→A）
- 非HTML内容（PDF/图片/视频链接）
- 需要JavaScript渲染的SPA页面
- 需要认证的页面（Cookie/Token）
- DNS解析失败
- 连接超时 vs 读取超时
- HTTPS证书问题

## 4. 安全测试

### 4.1 测试清单

| 编号 | 测试项 | 风险等级 | 验证方法 |
|------|--------|----------|----------|
| SEC-01 | SSRF防护 | 高 | 验证爬虫不能访问内网地址（127.0.0.1, 10.x, 192.168.x） |
| SEC-02 | XSS防护 | 高 | 爬取的HTML内容在展示时正确转义 |
| SEC-03 | SQL注入防护 | 高 | 搜索/过滤参数不能注入SQL |
| SEC-04 | API认证 | 高 | 未认证请求被拒绝 |
| SEC-05 | AI API Key保护 | 高 | Key不在前端暴露，不在日志中打印 |
| SEC-06 | 插件沙箱逃逸 | 高 | 恶意插件不能访问宿主系统 |
| SEC-07 | 路径遍历 | 中 | 文件操作不能访问预期目录外的文件 |
| SEC-08 | 速率限制 | 中 | API端点有请求频率限制 |
| SEC-09 | CORS配置 | 中 | 仅允许预期的域名访问 |
| SEC-10 | 依赖漏洞 | 中 | npm audit / snyk 无高危漏洞 |

### 4.2 爬虫安全专项

- 爬取目标的 robots.txt 遵守
- 爬取频率不构成 DDoS
- 不存储敏感个人信息
- 爬取内容的版权合规提示

## 5. 自动化测试方案

### 5.1 CI/CD 流水线

```yaml
# .github/workflows/test.yml 示意
name: Test Pipeline

on: [push, pull_request]

jobs:
  unit-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:unit -- --coverage
      - uses: codecov/codecov-action@v4

  integration-test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
      redis:
        image: redis:7
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:integration

  e2e-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npx playwright install
      - run: npm run test:e2e

  security-scan:
    runs-on: ubuntu-latest
    steps:
      - run: npm audit --audit-level=high
      - run: npx snyk test
```

### 5.2 测试命令规范

```json
{
  "scripts": {
    "test": "vitest run",
    "test:unit": "vitest run --project unit",
    "test:integration": "vitest run --project integration",
    "test:e2e": "playwright test",
    "test:coverage": "vitest run --coverage",
    "test:watch": "vitest watch"
  }
}
```

### 5.3 测试数据管理

- 使用 Factory 模式生成测试数据
- 集成测试使用独立测试数据库，每次测试前重置
- E2E 测试使用 seed 数据脚本
- 敏感数据使用 faker 生成

## 6. 质量保证流程

### 6.1 开发流程中的质量关卡

```
代码提交 → Pre-commit Hook（lint + format）
    ↓
Push → CI 单元测试 + 集成测试
    ↓
PR Review → 代码审查 + 测试覆盖率检查
    ↓
Merge → 完整测试套件 + E2E测试
    ↓
Release → 性能测试 + 安全扫描
```

### 6.2 质量指标监控

| 指标 | 目标 | 监控方式 |
|------|------|----------|
| 单元测试覆盖率 | ≥ 80% | Codecov |
| CI通过率 | ≥ 95% | GitHub Actions Dashboard |
| Bug修复时间 | P0 < 24h, P1 < 72h | Issue Tracker |
| 回归Bug率 | < 5% | 测试报告统计 |

### 6.3 Bug修复规范

每个bug修复必须包含：
1. 复现步骤文档
2. 根因分析
3. 回归测试用例
4. 修复验证记录

## 7. 潜在质量风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 网络依赖导致测试不稳定 | CI频繁失败 | Mock HTTP + 网络测试隔离标记 |
| AI API变更导致精炼失败 | 数据质量下降 | 契约测试 + 响应格式校验 + 降级策略 |
| 爬取目标网站结构变化 | 解析失败 | 插件化策略 + 监控告警 |
| 插件质量参差不齐 | 系统不稳定 | 沙箱隔离 + 插件审核机制 |
| 大规模数据下性能退化 | 用户体验差 | 定期性能测试 + 数据库索引优化 |
| 递归爬取失控 | 资源耗尽 | 深度限制 + 总量限制 + 超时机制 |
| 第三方依赖安全漏洞 | 安全风险 | 自动依赖更新 + 安全扫描 |
