# 个人信源库 — 产品需求文档

## 一、核心价值与用户场景

### 核心价值
用第一性原理来看，这个产品解决的根本问题是：**信息获取成本高、精炼耗时、知识沉淀困难**。

核心价值链：`信源发现 → 自动采集 → AI精炼 → 结构化入库 → 可检索复用`

### 典型用户场景

| 场景 | 用户画像 | 痛点 |
|------|---------|------|
| 技术跟踪 | 开发者，关注多个技术博客/文档站 | 手动逐站浏览效率低，信息分散 |
| 行业研究 | 分析师/创业者，监控行业动态 | 信息量大，难以快速提取关键洞察 |
| 内容创作 | 自媒体/写作者，需要素材积累 | 收藏夹堆积，缺乏结构化整理 |
| 学术追踪 | 研究人员，跟踪论文/报告 | 原始内容冗长，需要摘要和要点提取 |

## 二、功能需求细化

### P0 — 核心闭环（MVP必须）

**US-01 单页面抓取**
> 作为用户，我希望输入一个URL，系统自动抓取页面内容并存储，以便后续精炼和查阅。

验收标准：
- 输入URL → 抓取正文内容（去除导航/广告等噪音）
- 支持静态HTML页面
- 抓取结果持久化存储
- 抓取失败时给出明确错误提示

**US-02 AI精炼**
> 作为用户，我希望对抓取的原始内容调用AI进行摘要/提取/分类，得到结构化的精炼结果。

验收标准：
- 配置OpenAI兼容接口（endpoint + API key + model）
- 支持自定义精炼提示词模板
- 精炼结果与原始内容关联存储
- 精炼失败可重试

**US-03 结果预览**
> 作为用户，我希望在界面上查看信源的原始内容和精炼结果。

验收标准：
- 信源列表展示（标题、来源、时间、状态）
- 详情页展示原始内容 + 精炼结果
- 基本的搜索/筛选能力

**US-04 任务管理**
> 作为用户，我希望查看爬取和精炼任务的执行状态。

验收标准：
- 任务队列可视化（待执行/执行中/成功/失败）
- 失败任务可手动重试

### P1 — 能力扩展

**US-05 整站爬取**
> 作为用户，我希望输入一个站点入口URL，系统自动发现子页面并逐一抓取。

验收标准：
- 第一步：爬取入口页，解析出子页面链接列表
- 第二步：对每个子页面创建单页面抓取任务
- 支持配置爬取深度、URL过滤规则
- 支持配置并发数和请求间隔（礼貌爬取）

**US-06 定时任务**
> 作为用户，我希望配置定时爬取，自动监控信源更新。

**US-07 策略包机制**
> 作为用户，我希望针对特定网站使用定制的抓取策略（如登录态、动态渲染、特殊DOM结构）。

验收标准：
- 通用策略包：基于HTTP请求 + HTML解析，覆盖常规静态页面
- 插件接口：定义标准的策略包接口（抓取、解析、清洗）
- 策略包可按域名/URL模式匹配

### P2 — 体验优化

- **US-08** 批量导入URL
- **US-09** 标签/分类管理
- **US-10** 导出功能（Markdown/JSON）
- **US-11** 精炼模板市场（预置多种精炼策略）

## 三、MVP范围定义（修订版）

```
MVP = 整站爬取 + 任务系统 + 插件框架 + AI精炼 + 基础UI
```

**修订理由**：原MVP仅含单页面抓取，无法验证核心架构（任务递归生成、插件可扩展性）。

MVP验证假设：**用户愿意通过"抓取+AI精炼"的方式构建个人知识库，且系统架构能支撑可扩展的爬取策略**。

### MVP核心流程

1. 用户输入站点入口URL
2. 系统抓取入口页，解析出子页面链接
3. 为每个子页面创建抓取任务（任务递归生成）
4. 任务队列依次执行，调用插件抓取正文
5. 抓取完成后自动触发AI精炼
6. UI展示任务执行状态和精炼结果

### MVP包含

- 整站爬取（入口页 → 子页面列表 → N个单页任务）
- 任务系统（优先级队列、状态机、失败重试）
- 插件框架（接口定义 + 通用插件 + 域名匹配路由）
- AI精炼（单一策略：摘要+关键词提取）
- 基础UI（任务列表 + 信源详情页）

### MVP不包含

定时任务、标签分类、导出功能、多种精炼模板、搜索筛选、插件市场、热加载

### 4周实现计划

- Week 1：任务系统 + 通用插件（单页抓取能力）
- Week 2：整站爬取逻辑（链接提取 + 任务递归生成）
- Week 3：AI精炼集成 + 基础UI
- Week 4：联调测试 + 边界case处理

## 四、用户体验建议

1. **快速上手**：首次使用引导用户完成一次"输入URL → 抓取 → 精炼 → 查看结果"的完整流程，30秒内看到价值
2. **渐进式配置**：AI接口配置提供合理默认值，精炼模板预置通用方案，用户可以零配置开始使用
3. **状态透明**：任务执行过程实时反馈（抓取中/精炼中/完成），避免用户等待焦虑
4. **容错友好**：网页抓取天然不稳定，失败场景要给出可操作的提示（如"该页面需要登录"、"页面结构无法解析"），而非通用错误
5. **对比视图**：精炼结果页面同时展示原文和精炼结果，方便用户判断精炼质量

## 五、风险与挑战

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| 反爬机制 | 大量网站有反爬措施，抓取成功率不可控 | MVP先聚焦静态页面；P1引入策略包机制应对特定站点 |
| 内容解析质量 | 正文提取不准确（混入导航/广告/评论） | 使用readability算法做正文提取；策略包允许针对性优化 |
| AI精炼成本 | 大量内容精炼会产生API调用费用 | 支持配置本地模型（Ollama等）；提供精炼前预览，用户确认后再调用 |
| 法律合规 | 爬取内容可能涉及版权问题 | 定位为个人使用工具；仅存储精炼摘要而非全文复制；尊重robots.txt |
| 数据膨胀 | 长期使用后数据量增长 | SQLite对单用户场景足够；提供数据清理/归档功能 |
| 动态渲染页面 | SPA/JS渲染页面无法通过HTTP请求获取 | MVP明确不支持；P1通过Playwright等headless browser策略包解决 |
