# 个人信源库 — 产品需求文档

> 更新日期: 2026-03-17

## 一、核心价值与用户场景

**核心价值链**：`信源发现 → 自动采集 → AI精炼 → 结构化入库 → 通知推送 → 可检索复用`

从「人找信息」变为「信息找人」。

### 典型用户场景

| 场景 | 用户画像 | 痛点 |
|------|---------|------|
| 技术跟踪 | 开发者，关注多个技术博客/文档站 | 手动逐站浏览效率低，信息分散 |
| 行业研究 | 分析师/创业者，监控行业动态 | 信息量大，难以快速提取关键洞察 |
| 内容创作 | 自媒体/写作者，需要素材积累 | 收藏夹堆积，缺乏结构化整理 |
| 学术追踪 | 研究人员，跟踪论文/报告 | 原始内容冗长，需要摘要和要点提取 |

## 二、已实现功能

### 采集能力

- **整站爬取**：入口页 → 子页面发现 → 递归爬取，支持深度/数量/URL过滤控制
- **RSS 爬取**：解析 RSS/Atom feed，自动发现文章链接并逐篇爬取
- **单页爬取**：指定 URL 直接爬取
- **反爬处理**：UA 轮换、请求限速、并发控制

### 任务系统

- 优先级队列（asyncio.PriorityQueue）
- 任务状态机：pending → running → success/failed
- 失败自动重试（最多 3 次）
- APScheduler 定时触发（cron 表达式）
- 任务树可视化（父子任务关联）

### AI 精炼

- OpenAI 兼容接口（支持 DeepSeek、本地模型等）
- 自动生成：标题、摘要、关键词
- 质量评分（0-100 分），评分标准由分类配置决定
- 手动触发精炼 + 批量精炼

### 分类管理

- 分类 CRUD（名称、描述、颜色标签）
- 每个分类独立配置：
  - `refine_prompt_system`：AI 总结时的关注重点（系统提示词）
  - `quality_criteria`：质量评分标准（0-100 分维度定义）
- 预设模板：技术文档、投资资讯、阅读笔记
- 信源绑定分类，继承精炼配置

### 通知管理

- **通知渠道**（NotificationChannel）：Webhook、Telegram Bot、飞书 Webhook
- **通知规则**（NotificationRule）：绑定分类，支持三种触发条件：
  - `new_content`：有新精炼结果即触发
  - `quality_threshold`：质量评分达到阈值触发
  - `keyword_match`：关键词匹配触发
- **通知模式**：
  - 即时（instant）：每条结果立即发送
  - 聚合（batch）：时间窗口 + 数量上限，合并发送
- 飞书消息卡片格式（interactive）作为默认，支持纯文本降级
- 通知去重（同一结果+同一规则只发一次）
- 发送失败重试（3 次，指数退避）

### 前端界面

- 信源管理（CRUD、手动触发、定时配置）
- 分类管理（含通知规则 Tab）
- 任务列表（状态筛选、实时刷新）
- 采集结果（质量分数滑块筛选、多维排序）
- 精炼结果详情（原文 + 精炼内容对比）
- 通知渠道管理（CRUD、测试发送）

## 四、风险与挑战

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| 反爬机制 | 抓取成功率不可控 | 已实现 UA 轮换、限速；P1 引入 Playwright 处理 JS 渲染 |
| AI 精炼成本 | 大量内容产生 API 费用 | 支持本地模型（Ollama 等）；质量评分可过滤低价值内容 |
| 通知轰炸 | 批量精炼时推送过多 | 聚合模式 batch_window + batch_max_count 双重控制 |
| Token/密钥安全 | 渠道配置含敏感信息 | API 返回时脱敏（bot_token → `***`），仅创建/更新时接收明文 |
| 法律合规 | 爬取内容涉及版权 | 定位为个人使用工具；尊重 robots.txt |
